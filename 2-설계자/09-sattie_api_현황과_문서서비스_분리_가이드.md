# sattie_api 현황과 문서서비스 분리 가이드

## 문서 정보
- 최종 갱신일자: `03/14 15:46`

## 1. 문서 목적
- 현재 `src/app/sattie_api.py`의 실제 역할을 개발팀이 오해 없이 이해하도록 정리한다.
- 마크다운 문서 서비스 구현 시 `sattie_api.py`에서 무엇을 재사용하고 무엇을 분리해야 하는지 기준을 제공한다.
- 문서 서비스가 기존 위성영상 데모 기능의 부속 기능인지, 독립 서비스인지 구조적으로 구분한다.

## 2. 대상 파일
- `src/app/sattie_api.py`
- `src/app/core.py`
- `src/app/md_viwer.py`

## 3. 결론 요약
- `sattie_api.py`는 마크다운 문서 서비스 전용 파일이 아니다.
- 이 파일의 주역할은 위성영상 샘플 API, OSM 기반 모사 생성, mock store 관리, 데모 UI 제공이다.
- 마크다운 문서 서비스와 직접 관련 있는 부분은 `guide` 계열 라우트 일부뿐이다.
- 따라서 문서 서비스 MVP 구현은 `sattie_api.py`를 중심으로 확장하는 방식보다, 문서 서비스 모듈을 분리하고 필요한 부분만 재사용하는 방식이 맞다.

## 4. 현재 `sattie_api.py`의 실제 역할

### 4.1 서버 엔트리포인트
- FastAPI `app` 생성
- OpenAPI 메타데이터 설정
- 미들웨어 등록
- startup 시 카탈로그 초기화
- 정적 파일 마운트

### 4.2 위성영상 샘플 API
- `/health`
- `/images`
- `/images/{image_id}`
- `/images/{image_id}/download`
- `/images/{image_id}/content`

### 4.3 OSM 기반 위성영상 모사 기능
- `/osm/images/generate-all`
- `/osm/images/generate`
- `/osm/images`
- `/osm/images/items`
- `/osm/images/{request_id}`
- `/osm/images/{request_id}/{level}/download`
- `/osm/images/{request_id}/{level}/content`

### 4.4 운영/관리 기능
- `/admin/mock-store/info`
- `/admin/osm-store/info`
- `/admin/osm-store/delete`
- `/admin/mock-store/rebuild`
- `/admin/mock-store/delete`

### 4.5 데모 UI 기능
- `/ui`

- `/ui/static`

### 4.6 마크다운 문서 관련 기능
- `/guide`
- `/guide/{guide_path}`
- `/guide/upload`

## 5. 문서 서비스와 직접 관련 있는 부분

### 5.1 재사용 가능한 부분
- `guide` 폴더 내 Markdown 파일 목록 조회 흐름
- `.md` 업로드 검증의 기본 패턴
- Markdown 파일 읽기 후 HTML 렌더링하는 흐름
- `md_viwer.py`의 경량 렌더러 사용 방식

### 5.2 문서 서비스 전환 시 참고 가능한 함수
- `_guide_markdown_files()`
  - 현재 `guide/` 스캔 패턴 참고 가능
- `_sanitize_md_filename()`
  - 업로드 파일명 검증 패턴 참고 가능
- `_resolve_guide_file()`
  - 경로 traversal 방지 패턴 참고 가능
- `guide_view()`
  - Markdown 문서 읽기 후 HTML 반환 흐름 참고 가능
- `guide_upload()`
  - 업로드 후 파일 저장 흐름 참고 가능

### 5.3 직접 재사용보다 재설계가 필요한 부분
- `GUIDE_DIR` 단일 경로 전제
- `/guide` 기반 URL 구조
- 업로드 후 단순 리다이렉트 응답
- 메타데이터 인덱스 없음
- slug 기반 상세 접근 없음
- 검색, 카테고리, 태그, 관련 문서, SEO 메타 없음

## 6. 문서 서비스와 무관하거나 약한 관련만 있는 부분

### 6.1 무관한 기능
- 위성영상 샘플 카탈로그 관리
- OSM 타일 기반 이미지 생성
- TIFF/BIN 생성 및 다운로드
- mock store 재생성/삭제
- 위성영상 데모 UI

### 6.2 이유
- 이 기능들은 모두 위성영상 샘플 생성과 탐색을 위한 것이다.
- 문서 허브, 문서 상세, 검색, 메타데이터 인덱싱과 직접 연결되지 않는다.
- 문서 서비스 구현 시 그대로 품고 가면 책임 분리가 무너진다.

## 7. 개발팀 구현 기준

### 7.1 `sattie_api.py`를 어떻게 봐야 하는가
- 문서 서비스 메인 구현 파일로 보지 않는다.
- 기존 서비스 진입점이자 통합 조립 파일로 본다.
- 문서 서비스는 이 파일 안에 덧붙이는 방식이 아니라, 별도 모듈을 분리한 뒤 최종 연결만 이 파일에서 수행하는 방향이 맞다.

### 7.2 권장 구조
- `sattie_api.py`
  - 최종 앱 조립
  - 기존 위성영상/OSM/UI 라우트 유지
  - 문서 서비스 라우트 include 또는 연결
- 문서 서비스 전용 모듈
  - 라우트
  - 서비스
  - 인덱서
  - 모델
  - SEO

### 7.3 개발 우선순위
- 기존 `guide` 기능을 바로 확장하지 말고 문서 서비스 전용 라우트를 먼저 분리한다.
- 기존 `guide` 기능은 내부 참고 구현 또는 전환기 호환 기능으로만 본다.
- `sattie_api.py`는 문서 서비스 세부 로직을 계속 품지 않도록 유지한다.

## 8. 재사용 기준

### 8.1 유지 권장
- `md_viwer.py`의 렌더링 자산
- 파일 업로드 시 확장자 검증 패턴
- 경로 보안 검증 패턴

### 8.2 수정 후 재사용
- `guide/` 단일 폴더 스캔 로직
- 업로드 저장 경로 처리
- 렌더링 결과를 HTML 페이지로 조립하는 방식

### 8.3 재사용 비권장
- `/guide` URL 체계 자체
- 현재 `guide_upload()` 응답 계약
- 문서 서비스와 무관한 전역 상태 구조

## 9. 설계 판단
- `sattie_api.py`는 현재 문서 서비스보다 위성영상 데모 서비스에 더 강하게 결합되어 있다.
- 따라서 문서 서비스 MVP는 이 파일을 중심으로 커지면 안 된다.
- 문서 서비스는 독립 모듈로 분리하고, `sattie_api.py`는 최종 조립 레이어 역할만 하도록 가는 것이 맞다.
- 개발팀은 `guide` 기능을 “최종 제품 구조”가 아니라 “재사용 가능한 원형 구현”으로 해석해야 한다.

## 10. 권장 대안

### 10.1 권장 대안: 통합 앱 재구성
- 문서 서비스만 별도 앱으로 떼는 방식보다, 통합 앱 구조를 재구성하는 방식을 우선 권장한다.
- 이유는 현재 저장소가 이미 위성영상 서비스, OSM 모사 기능, UI, guide 기능을 하나의 FastAPI 앱으로 운영하고 있기 때문이다.
- 따라서 기존 엔트리포인트를 즉시 폐기하기보다, 라우트와 책임을 기능별 모듈로 나누고 최종 앱에서 조립하는 방향이 가장 현실적이다.

### 10.2 목표 구조
- `main.py`
  - 최종 FastAPI 앱 생성
  - 공통 미들웨어, startup, mount, include 조립
- `satellite_routes.py`
  - 위성영상 샘플 API
- `osm_routes.py`
  - OSM 생성/조회 API
- `admin_routes.py`
  - mock store 및 운영용 API
- `guide_routes.py`
  - 기존 guide 호환 라우트
- `docs_service/`
  - 문서 서비스 전용 라우트, 서비스, 인덱서, 모델, SEO

### 10.3 단계별 전환 방식
1. `sattie_api.py` 내부 라우트를 기능별 파일로 먼저 분리한다.
2. 새 통합 엔트리포인트가 각 라우트를 조립하도록 바꾼다.
3. 문서 서비스는 `docs_service` 모듈로 독립 구현한다.
4. 기존 `guide` 기능은 전환기 호환 기능으로 유지한다.
5. 최종적으로 `sattie_api.py`의 책임이 모두 다른 모듈로 이동하면 엔트리포인트 이름 변경 또는 제거를 검토한다.

### 10.4 이 대안을 권장하는 이유
- 기존 서비스 회귀 위험을 줄일 수 있다.
- 문서 서비스와 위성영상 데모 기능을 동시에 유지할 수 있다.
- 엔트리포인트를 단번에 없애지 않고 점진 분리가 가능하다.
- 개발팀이 기능 단위로 병렬 작업하기 쉽다.

### 10.5 개발팀 구현 기준
- `sattie_api.py`를 계속 키우지 않는다.
- 새 기능은 직접 추가하지 말고 기능별 모듈로 분리해 넣는다.
- 최종 앱은 조립 레이어만 담당하게 만든다.
- 문서 서비스는 기존 `guide` 라우트 확장이 아니라 독립 모듈로 구현한다.

## 11. 개발팀 전달용 한 줄 기준
- `sattie_api.py`는 문서 서비스 전용 파일이 아니며, 문서 서비스 관련 코드는 `guide` 계열 일부만 해당하므로 통합 앱 재구성 방식으로 기능별 모듈 분리를 우선 진행한다.
