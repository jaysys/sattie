## 4. 쟁점별 답변

### 4.1 운영자 화면 데이터 계약
- MVP에서는 별도 운영자 조회 API를 추가하지 않는다.
- `/admin/docs`는 서버 렌더링 화면으로 처리한다.
- 운영자 화면 데이터는 서버가 인덱스 기준으로 직접 조합해 템플릿에 주입한다.
- 운영자 화면에서 필요한 데이터는 아래로 고정한다.
  - `total_docs`
  - `published_docs`
  - `hidden_docs`
  - `last_indexed_at`
  - `items`
- `items`는 최소 아래 필드를 포함한다.
  - `title`
  - `slug`
  - `category`
  - `is_public`
  - `updated_at`
  - `source_path`
- 별도 JSON API는 후속 확장 범위다.

### 4.2 운영자 기능 범위
- MVP 운영자 화면은 읽기 전용 상태 조회와 전체 재색인만 포함한다.
- `대표문서 지정`은 MVP 제외다.
- 운영자 화면에서 허용하는 기능은 아래로 제한한다.
  - 인덱스 상태 확인
  - 공개 상태 확인
  - 전체 재색인 실행
- 공개 여부 변경, 카테고리 수정, 태그 수정, 대표문서 지정은 후속 확장 범위다.
- 개발팀은 운영자 화면을 편집 화면이 아니라 읽기 전용 운영 화면으로 구현하면 된다.

### 4.3 업로드 후 실패 처리 원칙
- MVP에서는 파일 저장과 재색인을 하나의 완전 원자 작업으로 보지 않는다.
- 파일 저장 성공 후 재색인 실패는 부분 성공으로 허용한다.
- 우선순위는 완전 롤백보다 문서 유실 방지에 둔다.
- 이 경우 정책은 아래와 같이 고정한다.
  - 파일은 저장 유지
  - 인덱스 반영 실패는 별도 실패 상태로 기록
  - 사용자에게는 업로드는 완료되었지만 목록 반영이 지연될 수 있다는 메시지를 제공
  - 운영 화면 또는 로그에서 재색인 필요 상태를 확인 가능하게 한다
- 따라서 MVP 기본 정책은 롤백이 아니라 부분 성공 허용이다.
- 구현 시 API 응답은 최소한 사용자 메시지와 운영 확인 경로를 보장해야 한다.

### 4.4 문서 스캔 범위와 업로드 저장 위치
- 업로드 저장 루트는 `uploads/docs/`로 고정한다.
- `guide/`, `docs/`는 기존 시드 문서 소스이며 읽기 전용 기준으로 본다.
- 스캔 대상 루트는 아래 3곳으로 고정한다.
  - `guide/`
  - `docs/`
  - `uploads/docs/`
- slug 충돌 시 특정 경로가 우선해 기존 문서를 덮어쓰지 않는다.
- 기존 slug는 유지하고, 후속 충돌 문서에 suffix를 부여한다.
- `source_path`는 실제 스캔된 상대경로를 그대로 저장한다.

### 4.5 검색 점수 규칙
- MVP에서는 QA 재현 가능 수준까지 숫자 가중치를 고정한다.
- 가중치는 아래와 같이 확정한다.
  - `title`: 10
  - `tags`: 7
  - `summary`: 5
  - `headings`: 4
  - `body`: 2
- 동일 문서가 여러 필드에 일치하면 점수는 합산한다.
- `matched_fields`는 기여 점수가 높은 필드 순서로 반환한다.
- 최종 정렬은 전체 점수 계산 후 적용한다.
- 동점이면 `updated_at` 최신순을 우선한다.
- 그 다음에 페이지네이션을 적용한다.

## 5. 개발팀 전달 요약
- 운영자 화면은 별도 조회 API 없이 서버 렌더링으로 구현한다.
- 운영자 기능은 조회 + 재색인까지만 MVP 범위다.
- 업로드는 부분 성공을 허용하며, 파일 저장 후 재색인 실패 시 파일은 유지한다.
- 업로드 저장 루트는 `uploads/docs/`로 고정한다.
- 스캔 대상은 `guide/`, `docs/`, `uploads/docs/`다.
- 검색 점수는 `title 10 / tags 7 / summary 5 / headings 4 / body 2`로 구현한다.

## 6. 구현 시 유의사항
- 실제 소스 구조가 `src/` 아래로 재배치되어 있더라도 본 문서의 설계 의도는 유지한다.
- 구현 경로 해석은 현재 저장소 구조를 기준으로 하되, 기능 기준과 우선순위는 본 문서와 기준 문서를 따른다.
- 기준 문서와 실제 코드 구조가 충돌하면 설계 의도를 임의 변경하지 말고 차이를 기록한 뒤 판단을 요청한다.
