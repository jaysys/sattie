# 설계 검토 의견

- 문서 최종 갱신일자: 03/14 15:47

## 1. 목적
- 설계 패키지와 설계 검토 답변서를 기준으로 현재 개발 착수 가능 여부를 판단한다.
- 개발팀이 구현 전에 추가 확인이 필요한 쟁점이 해소되었는지 기록한다.

## 3. 판단 결과
- 현재 설계 내용은 개발 착수 가능한 수준이라고 판단한다.
- 기존 검토에서 제기했던 구현 해석 여지는 `08-설계검토_답변서.md`에서 구현 기준으로 충분히 보완되었다.
- 따라서 현시점에서 개발팀이 추가 해석 없이 착수 가능한 상태다.

## 4. 착수 가능하다고 판단한 근거
### 4.1 전체 구현 기준이 이미 문서에 정리되어 있다
- MVP 범위가 고정되어 있다.
- HTML 라우트와 JSON API 구조가 고정되어 있다.
- 데이터 모델, 응답 형식, 화면 기준, 상태 전이, QA 기준이 문서로 분리되어 있다.
- 구현 순서와 완료 기준도 인수인계 문서에 정리되어 있다.

### 4.2 기존 쟁점은 답변서로 보완되었다
- 운영자 화면 데이터 계약
  - `/admin/docs`는 별도 조회 API 없이 서버 렌더링으로 처리한다고 확정되었다.
- 운영자 기능 범위
  - MVP 범위는 읽기 전용 상태 조회 + 전체 재색인으로 확정되었다.
  - `대표문서 지정`은 MVP 제외로 정리되었다.
- 업로드 실패 처리 원칙
  - 파일 저장 후 재색인 실패는 부분 성공 허용으로 확정되었다.
  - 파일은 유지하고, 사용자 메시지와 운영 확인 경로를 제공하는 기준이 정리되었다.
- 문서 스캔 범위와 저장 위치
  - 업로드 저장 루트는 `uploads/docs/`로 확정되었다.
  - 스캔 대상은 `guide/`, `docs/`, `uploads/docs/`로 확정되었다.
  - slug 충돌 시 후속 문서에 suffix를 부여하는 규칙이 정리되었다.
- 검색 점수 규칙
  - `title 10 / tags 7 / summary 5 / headings 4 / body 2`로 가중치가 고정되었다.
  - 동점 처리와 페이지네이션 적용 순서도 정리되었다.

## 5. 개발팀 유의사항
- 실제 구현 경로는 현재 저장소 구조상 `src/` 기준으로 해석하면 된다.
- 설계 의도는 설계 문서와 답변서를 우선 기준으로 유지해야 한다.
- 구현 중 설계 문서와 코드 구조가 충돌하면 임의 변경하지 말고 차이를 기록한 뒤 판단을 요청해야 한다.

## 6. 결론
- 설계 패키지와 답변서를 함께 기준으로 보면 개발 착수 가능 상태다.
- 추가 선결정이 필요한 미해결 설계 이슈는 현재 기준으로는 없다고 판단한다.
