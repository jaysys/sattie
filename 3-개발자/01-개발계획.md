# 개발계획

- 문서 최종 갱신일자: 03/14 15:47

## 1. 목적
- `2-설계자/` 폴더의 설계 문서를 기준으로 MVP 개발 순서를 확정한다.
- 구현, 검증, 이슈 정리를 모두 `3-개발자/` 폴더 안에서 관리한다.
- `1-기획자/`, `2-설계자/` 폴더는 읽기 전용으로 유지한다.
- 개발용 소스는 루트 아래 `src/` 폴더를 기준 작업 위치로 사용한다.
- 개발 착수 전에는 구현 대신 개발관리계획과 개발일정을 먼저 고정한다.

## 2. 기준 문서
- `2-설계자/01-마크다운_뷰어_웹서비스_설계서.md`
- `2-설계자/02-API_응답_명세_부록.md`
- `2-설계자/03-화면별_와이어프레임_컴포넌트_설계서.md`
- `1-기획자/01-마크다운_뷰어_웹서비스_기획서.md`

## 3. 현재 구현 기준
- 서버는 FastAPI 기반이며 현재 실행 및 구현 기준 소스는 `src/app/` 아래에 위치한다.
- 현재 주요 구현 파일은 `src/app/core.py`, `src/app/sattie_api.py`, `src/app/md_viwer.py`, `src/app/ui/`다.
- 운영/참고 문서는 `src/docs/`, `src/guide/` 아래로 정리되었다.
- 설계서 기준으로 문서 서비스 전용 라우트, 문서 인덱스 구조, 검색/업로드/운영 화면이 MVP 핵심 범위다.
- 현재 저장소 운영 원칙상 문서 산출물은 `3-개발자/`, 개발 소스 변경은 `src/` 하위에서 수행한다.
- 현재 시점의 우선순위는 구현 착수보다 관리 기준 정리이며, 실제 개발 시작 여부는 별도 승인 후 결정한다.

## 4. 개발 원칙
- 설계 문서의 MVP 범위와 단계별 우선순위를 그대로 따른다.
- 작업 시작 전에 `developer.md`의 역할과 권한 범위를 자가점검한다.
- 개발 진행 중 생성되는 계획, 메모, 검증 기록은 모두 `3-개발자/`에 작성한다.
- `1-기획자/`, `2-설계자/` 문서는 읽기만 하고 수정하지 않는다.
- 설계와 실제 코드가 충돌하면 원문 대신 `3-개발자/`에 차이를 기록한다.
- 신규 문서는 모두 `01-xxxx.md` 형식의 번호 접두사를 사용한다.
- 설계자와의 의사소통은 문서 기준으로 남기고, 프롬프트 답변은 요약만 한다.

## 5. 개발관리 기준
### 5.1 관리 원칙
- 구현 판단은 설계 문서와 답변서를 우선 기준으로 한다.
- 구현 전에 계획, 일정, 이슈, 검증 기준을 먼저 문서화한다.
- MVP 범위를 벗어나는 기능은 착수하지 않는다.
- 설계와 구현이 충돌하면 코드에서 임의 변경하지 않고 문서에 기록 후 판단을 받는다.
- 각 단계는 완료 기준과 검증 결과를 확인한 뒤 다음 단계로 넘긴다.

### 5.2 관리 대상
- 범위
  - `/hub`, `/p/{slug}`, `/upload`, `/admin/docs`
  - `/api/docs/list`
  - `/api/docs/search`
  - `/api/docs/{slug}`
  - `/api/docs/{slug}/raw`
  - `POST /api/docs/upload`
  - `POST /api/docs/reindex`
- 소스 위치
  - 개발용 소스 변경은 `src/` 하위에서만 수행한다.
- 문서 위치
  - 개발관리 산출물은 `3-개발자/` 하위에서만 작성한다.

### 5.3 변경관리 기준
- 아래 항목은 변경 요청 대상으로 본다.
  - MVP 범위 변경
  - API 응답 필드 추가/삭제/이름 변경
  - 화면 흐름 변경
  - 저장 경로 및 인덱스 정책 변경
  - 운영자 기능 범위 변경
- 변경 요청이 발생하면 다음 순서로 처리한다.
  1. 변경 사유 문서화
  2. 영향 범위 식별
  3. 설계 기준과 차이 기록
  4. 승인 전에는 구현 보류

### 5.4 품질관리 기준
- API는 설계 부록과 필드, 상태코드, 정렬 규칙이 맞아야 한다.
- 화면은 와이어프레임, 상태전이, 접근성 기준을 충족해야 한다.
- 데이터 모델은 데이터 사전과 일치해야 한다.
- QA 체크리스트 항목을 기준으로 완료 여부를 판정한다.
- 기존 `/guide`, `/ui`, 위성영상 API에 회귀가 없어야 한다.

### 5.5 일정관리 기준
- 일정은 `3-개발자/03-개발일정.md`를 기준으로 관리한다.
- 단계 완료 기준을 충족하지 못하면 다음 단계로 넘어가지 않는다.
- 일정 지연이 생기면 원인을 기능 범위, 설계 충돌, 구현 난이도, 검증 이슈 중 하나로 분류해 기록한다.

## 6. 단계별 개발 계획
### 1단계. 개발관리 기준 고정
- `3-개발자/03-개발일정.md`를 기준으로 단계, 선후관계, 완료 게이트를 확정한다.
- 구현 착수 전 승인 조건과 산출물 작성 방식을 먼저 맞춘다.

### 2단계. 문서 서비스 기반 정리
- 문서 서비스 전용 라우트 구조를 분리한다.
- 문서 메타데이터, slug, front matter, 인덱스 구조를 설계서 기준으로 정리한다.
- 파일 시스템 + JSON 인덱스 기반 MVP 구조를 우선 구현한다.
- `src/` 기준 실행 구조와 문서 저장 경로를 설계 의도와 충돌 없이 정리한다.

### 3단계. 문서 API 구현
- `GET /api/docs/list`
- `GET /api/docs/search`
- `GET /api/docs/{slug}`
- `GET /api/docs/{slug}/raw`
- `POST /api/docs/upload`
- `POST /api/docs/reindex`
- API 응답 형식은 `2-설계자/API_응답_명세_부록.md`와 정합성을 맞춘다.

### 4단계. 화면 구현
- 랜딩 페이지
- 문서 허브 페이지
- 문서 상세 페이지
- 업로드 페이지
- 운영자 문서 관리 페이지
- 화면 구조와 컴포넌트 경계는 `2-설계자/화면별_와이어프레임_컴포넌트_설계서.md`를 기준으로 구현한다.

### 5단계. 품질 보강
- 검색 정확도와 정렬 규칙 보강
- SEO 메타 및 공유 카드 반영
- 렌더링 안전성, 업로드 검증, 에러 처리 보강
- 모바일 대응, 접근성, 회귀 테스트 수행
