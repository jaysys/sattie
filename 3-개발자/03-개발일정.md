# 개발일정

- 문서 최종 갱신일자: 03/14 15:47

## 1. 목적
- 구현 착수 전에 단계별 일정, 선후관계, 완료 기준을 고정한다.
- 실제 착수 시 작업 순서와 검증 순서를 동일 기준으로 관리한다.

## 2. 일정 수립 기준
- 달력 날짜가 아니라 상대 일정 기준으로 관리한다.
- `D1`은 구현 착수 승인일을 의미한다.
- 각 단계는 선행 단계 완료와 검증 통과를 조건으로 시작한다.
- `1단계 ~ 4단계`를 MVP 개발 구간으로 본다.
- 사용자는 MVP 완료 시점에 실제 서비스를 띄워 확인한다.

## 3. 전체 일정표
| 단계 | 기간 | 구분 | 개발 기능/작업 | 완료 기준 |
|---|---|---|---|---|
| 준비 단계 | `D-Pre` | 착수 준비 | 설계 검토 완료, 개발계획 확정, 개발일정 확정 | 개발 착수 전 관리 문서가 `3-개발자/`에 정리됨 |
| 1단계 | `D1 ~ D2` | MVP 기본 골격 | 문서 서비스 모듈 구조, 기본 라우트 연결, 실행 구조 정리 | `/hub`, `/p/{slug}`, `/upload`, `/api/docs/*` 기본 진입 구조 준비, 기존 `/guide`, `/ui`와 구조 충돌 없음 |
| 2단계 | `D3 ~ D4` | MVP 인덱서/메타 | 스캐너, front matter 파서, slug 생성기, heading 추출기, `documents.json`, `search.json` 생성기 | `guide/`, `docs/`, `uploads/docs/` 스캔 가능, 메타데이터와 검색 인덱스 생성 가능 |
| 3단계 | `D5 ~ D6` | MVP JSON API | 목록 API, 검색 API, 상세 API, 원문 API, 업로드 API, 재색인 API | API 응답이 설계 부록과 답변서 기준 만족, 업로드 부분 성공 정책 반영 |
| 4단계 | `D7 ~ D8` | MVP HTML 화면 | 랜딩, 허브, 상세, 업로드, 운영자 화면 | 와이어프레임과 상태전이 기준 충족, 운영자 화면 서버 렌더링 동작 |
| MVP 완료 확인 | `D8` | 확인 게이트 | 서비스 기동, 핵심 사용자 흐름 확인, 화면/라우트 실사용 점검 | 사용자가 서비스에 접속해 MVP 범위를 직접 확인 가능 |
| 5단계 | `D9 ~ D10` | 품질 보강 | 검색 정렬 보강, 에러 처리 통일, SEO 메타 반영, 반응형/접근성/회귀 검증 | QA 체크리스트 핵심 항목 통과, `/guide`, `/ui`, 기존 API 회귀 없음 |

## 4. MVP 개발 구간
| 구간 | 포함 단계 | 목표 |
|---|---|---|
| MVP 개발 구간 | 1단계 ~ 4단계 | 문서 허브 서비스의 핵심 기능을 실제 확인 가능한 상태까지 구현 |
| MVP 완료 확인 | `D8` | 사용자가 서비스를 띄워 `/hub`, `/p/{slug}`, `/upload`, `/admin/docs`, `/api/docs/*` 흐름을 직접 점검 |
| 후속 보강 구간 | 5단계 | 품질 보강, SEO, 접근성, 회귀 검증 |

## 5. 단계별 상세 일정
### 4.1 준비 단계
| 항목 | 내용 |
|---|---|
| 기간 | `D-Pre` |
| 목표 | 설계 검토 완료, 개발관리 기준 확정, 개발일정 확정 |
| 산출물 | `01-개발계획.md`, `02-설계검토의견.md`, `03-개발일정.md` |
| 종료 조건 | 구현 착수 전 필요한 문서 기준이 모두 정리됨 |

### 4.2 1단계. 문서 서비스 기본 골격
| 기능 | 내용 | 종료 조건 |
|---|---|---|
| 문서 서비스 모듈 구조 | 문서 서비스 로직을 기존 위성영상 데모와 분리할 구조 준비 | 독립 모듈 구조가 정리됨 |
| 기본 라우트 연결 | `/hub`, `/p/{slug}`, `/upload`, `/api/docs/*` 진입 구조 준비 | 플레이스홀더 또는 기본 응답이라도 정상 진입 |
| 실행 구조 정리 | `src/` 기준 실행 경로와 라우트 연결 방식 정리 | 기존 `/guide`, `/ui`와 충돌 없음 |

### 4.3 2단계. 인덱서 및 메타데이터
| 기능 | 내용 | 종료 조건 |
|---|---|---|
| 스캐너 | `guide/`, `docs/`, `uploads/docs/` 스캔 | 대상 문서 목록 안정 반환 |
| front matter 파서 | `title`, `summary`, `category`, `tags`, `is_public`, `order` 추출 | front matter 유무와 무관하게 메타 생성 |
| slug 생성기 | 파일명 기반 slug 생성과 충돌 처리 | 충돌 시 suffix로 해결 |
| heading 추출기 | H2~H3 목차 추출 | 상세/TOC에 재사용 가능한 배열 생성 |
| 인덱스 생성기 | `documents.json`, `search.json` 생성 | 전체 재생성 가능, fallback 전략 유지 |

### 4.4 3단계. JSON API 구현
| 기능 | 경로 | 종료 조건 |
|---|---|---|
| 목록 API | `GET /api/docs/list` | `category`, `tag`, `sort`, `page`, `page_size` 처리 |
| 검색 API | `GET /api/docs/search` | `title`, `tags`, `summary`, `headings`, `body` 검색 가능 |
| 상세 API | `GET /api/docs/{slug}` | `html`, `headings`, `related_documents` 포함 |
| 원문 API | `GET /api/docs/{slug}/raw` | 원본 Markdown 반환 |
| 업로드 API | `POST /api/docs/upload` | 저장, slug, public_url 반환, 부분 성공 정책 반영 |
| 재색인 API | `POST /api/docs/reindex` | 전체 재색인 및 문서 수 반환 |

### 4.5 4단계. HTML 화면 구현
| 기능 | 내용 | 종료 조건 |
|---|---|---|
| 랜딩 페이지 | 허브 진입, 업로드 유도, 서비스 가치 제시 | 주요 진입 CTA 동작 |
| 문서 허브 페이지 | 문서 카드, 카테고리, 검색, 정렬 | 허브 탐색 흐름 성립 |
| 문서 상세 페이지 | 본문, 메타, TOC, 관련 문서, 공유, 원문 링크 | 상세 읽기 흐름 성립 |
| 업로드 페이지 | 파일 선택, 업로드 상태, 완료 후 이동 | 업로드 상태전이 기준 충족 |
| 운영자 화면 | 문서 수, 공개 상태, 재색인 | 서버 렌더링 + 읽기 전용 운영 화면 동작 |

### 4.6 5단계. 품질 보강 및 회귀 검증
| 기능 | 내용 | 종료 조건 |
|---|---|---|
| 검색 품질 | 정렬/점수 규칙 보강 | 답변서 기준 가중치 반영 |
| 에러 처리 | API/HTML 오류 처리 통일 | 의미 있는 메시지 제공 |
| SEO 메타 | title, description, canonical, 공유 메타 | 상세 페이지 메타 반영 |
| 반응형/접근성 | 모바일, 태블릿, 키보드, 라벨 점검 | QA 기준 충족 |
| 회귀 검증 | `/guide`, `/ui`, 기존 API 영향 점검 | 회귀 없음 |

## 6. 선후관계
- 준비 단계 완료 후 1단계 시작
- 1단계 완료 후 2단계 시작
- 2단계 완료 후 3단계 시작
- 3단계 완료 후 4단계 시작
- 4단계 완료 후 MVP 완료 확인 수행
- MVP 완료 확인 후 5단계 시작

## 7. 병렬 가능 범위
- 문서 초안 병렬 작성
  - 검증기록 양식
  - 이슈메모 양식
- 구현 병렬 가능 범위
  - 2단계 일부 메타 추출 로직과 4단계 화면 마크업 초안
- 병렬 비권장 범위
  - 인덱서 완료 전 검색 API 구현
  - 상세 응답 구조 고정 전 상세 화면 확정

## 8. 단계 게이트
- 1단계 게이트
  - 구조 분리와 라우트 진입 확인
- 2단계 게이트
  - 스캔, slug, heading, 인덱스 파일 생성 확인
- 3단계 게이트
  - API 응답 형식과 오류 처리 정합성 확인
- 4단계 게이트
  - 화면 흐름, 상태전이, 운영자 화면 확인
- MVP 완료 확인 게이트
  - 사용자가 서비스에 접속해 핵심 MVP 흐름을 직접 확인
  - `/hub`, `/p/{slug}`, `/upload`, `/admin/docs`, `/api/docs/*` 동작 확인
- 5단계 게이트
  - QA, 회귀, 접근성, 반응형 확인

## 9. 일정 운영 원칙
- 각 단계 종료 시 완료 여부를 문서로 남긴다.
- 지연이 발생하면 다음 단계로 밀지 않고 원인을 먼저 기록한다.
- 일정 단축을 위해 설계 해석을 생략하지 않는다.
- 구현 착수 승인 전에는 `D1`을 시작하지 않는다.

## 10. 현재 상태
- 현재는 준비 단계다.
- 개발관리계획과 개발일정을 먼저 정리하는 중이다.
- 실제 구현 단계는 아직 시작하지 않았다.
